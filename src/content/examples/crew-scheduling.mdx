## Problem Description

An airline must assign crews to a set of flight legs. Each crew member starts and ends their duty period at the same **base** airport. A **pairing** is a sequence of flight legs forming a round trip from a crew member's base. Everything is self-contained: this example does not build on any other example on this site.

## Set-Partitioning Formulation

The crew scheduling master formulation is a **set-partitioning** problem: every flight leg must be covered by **exactly one** pairing:

<MathBlock
  formula={`
\\min \\quad & \\sum_{b=1}^{B} \\sum_{p \\in P^b} c_p x_p \\\\
\\text{s.t.} \\quad & \\sum_{b=1}^{B} \\sum_{p \\in P^b} a_{fp} x_p = 1 \\quad \\forall f \\in F \\quad \\text{(coupling: coverage constraints)} \\\\
& \\sum_{p \\in P^b} x_p \\ge 1 \\quad \\forall b = 1, \\ldots, B \\quad \\text{(sub-problem: base feasibility)} \\\\
& x_p \\in \\{0,1\\} \\quad \\forall p
`}
  display={true}
/>

<Callout type="info">
**Coupling constraints**: The flight-leg coverage constraints <InlineMath formula="\sum_{b,p} a_{fp} x_p = 1" /> are the **coupling constraints** — they link pairings from all bases and must hold globally.

**Sub-problem constraints**: The constraint <InlineMath formula="\sum_{p \in P^b} x_p \ge 1" /> is a **per-base sub-problem constraint**, ensuring each base deploys at least one pairing. Crucially, pairings from different bases are otherwise independent — structure enabling decomposition.

</Callout>

## Dantzig-Wolfe Decomposition and Column Generation

The LP relaxation has one variable per feasible pairing. There are exponentially many pairings in <InlineMath formula="P^b" /> for each base <InlineMath formula="b" />, making explicit enumeration infeasible.

Dantzig-Wolfe Decomposition decomposes along bases: relax the coupling coverage constraints with dual variables <InlineMath formula="\pi_f" />.

**Dual variables**: <InlineMath formula="\pi_f" /> is the marginal cost saving of covering flight <InlineMath formula="f" /> from the master. A pairing is attractive if its cost (duty time plus overheads) is less than the total dual value of the legs it covers.

**Pricing sub-problem for base <InlineMath formula="b" />**: Find the minimum reduced-cost pairing from base <InlineMath formula="b" />:

<MathBlock formula={`\\bar{c}_p = c_p - \\sum_{f \\in F} \\pi_f^* a_{fp}`} display={true} />

<MathBlock
  formula={`
\\min \\quad & c_p - \\sum_{f \\in F} \\pi_f^* a_{fp} \\\\
\\text{s.t.} \\quad & p \\in P^b \\quad \\text{(feasible pairing from base } b\\text{)}
`}
  display={true}
/>

This sub-problem is typically solved as a **constrained shortest-path** problem on a time-space network for base <InlineMath formula="b" />, incorporating duty-time rules, rest requirements, and airport curfews.

If <InlineMath formula="\min_p \bar{c}_p < 0" /> for any base, a new improving pairing (column) is added to the restricted master problem (RMP).

## Column Generation for Crew Pairings

The algorithm iterates:

1. **Solve the RMP** (LP relaxation over the current subset of pairings).
2. **Extract duals** <InlineMath formula="\pi_f^*" /> from coverage constraints.
3. **For each base <InlineMath formula="b" />**: solve the pricing sub-problem to find the most attractive new pairing.
4. **If any pairing has negative reduced cost**: add it to the RMP and go to step 1.
5. **Otherwise**: the LP relaxation is optimal. Apply branch-and-price for integrality if needed.

<Callout type="tip">
  In practice, only a tiny fraction of all feasible pairings ever enter the master problem. Column
  generation makes what would otherwise be an intractable billion-variable IP into a
  routinely-solved industrial problem — airline crew scheduling software uses exactly this
  technique.
</Callout>
